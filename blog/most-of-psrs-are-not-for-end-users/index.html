Most of PSRs are not for end users
Mar 7, 2017
2 minutes read

If you’re professionally developing with PHP, you should be aware about PSRs — standard recommendations put together by prominent project representatives.

Even among people who are aware of these there is a common mistunderstanding about the purpose and applicability of PSRs. This is, of course, my personal view on them but I know a thing or two since I’m Yii representative in PHP-FIG.

So the current PSRs could be divided into the following groups:

  • Coding standards (PSR-1, PSR-2, PSR-5, PSR-12).
  • Early interfaces (PSR-3, PSR-6).
  • Later interfaces (PSR-7, PSR-13, PSR-11, PSR-14, PSR-15, PSR-16, PSR-17).
  • Autoloading (PSR-0, PSR-4).
  • Procedures (PSR-9, PSR-10).

Coding Standards

The purpose and applicability of coding standards is a bit different in OpenSource applications and closed code applications. In OpenSource these are to make overall PHP world code style uniform. In order to be successful you have to use style which is close to at least PSR-2. In closed source projects these standard recommendations could be either used as is or as a base for project team’s own standard. Not strictly following PSR is OK as long as team has its own standard written and enforced.

Early Interfaces

In the beginning PHP-FIG was a small group of framework developers with the goal to be able to reuse and interchange parts of their frameworks. That is why early interfaces focus exactly on that and less on user experience or additional features. A common misunderstanding is that PSR interface should be exposed to end users as is. Really there’s no such requirement. Framework may keep its own interfaces more suitable for end users while PSR interfaces are used under the hood to actually do the job.

Later interfaces

Above statement about not exposing interfaces to end users is less valid with later interfaces. PHP-FIG goal was adjusted multiple times and was moved from framework interoperability to interfaces actually used by end users.

Autoloading

Unlike interfaces, autoloading is strictly practical end user thing.
